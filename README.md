# Локальная кольцевая «почта» (вариант 18)

Программа передачи сообщений между тремя узлами в кольцевой топологии с реализацией **физического**, **канального** и **прикладного** уровней (курс «Сетевые технологии в АСОИУ», МГТУ им. Баумана).

---

## Запуск на Windows

### 1. Установка Python

1. Скачайте **Python 3.10+** с [python.org](https://www.python.org/downloads/windows/).
2. При установке отметьте **«Add python.exe to PATH»** и **«tcl/tk and IDLE»** (нужен для графического интерфейса).
3. Проверьте в **cmd** или **PowerShell**:

```cmd
python --version
```

### 2. Подготовка проекта

Откройте терминал в папке проекта (например, `C:\Users\Имя\Documents\Курсач`):

```cmd
cd C:\Users\Имя\Documents\Курсач
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

После активации в начале строки появится `(.venv)`.

### 3. Демонстрация на одном ПК (рекомендуется)

Один запуск — три окна почты, эмуляция кольца RS-232 без кабелей:

```cmd
python demo.py
```

1. Дождитесь статуса **«Подключено»** во всех трёх окнах.
2. **Почта → Написать письмо** — выберите адресата (узел 2 или 3).
3. У получателя — двойной щелчок по письму → **Отметить прочитанным**.
4. У отправителя во **Исходящих** статус сменится на **«прочитано»**.

### 4. Три отдельных окна (три терминала)

В трёх окнах **cmd**, в каждом активируйте venv и запустите:

```cmd
python main.py --node 1 --demo-ring
python main.py --node 2 --demo-ring
python main.py --node 3 --demo-ring
```

Флаг `--demo-ring` подключает общий программный симулятор кольца (как в `demo.py`).

### 5. Реальные COM-порты (com0com, по ТЗ)

Для защиты с «настоящими» портами:

1. Установите [com0com](https://sourceforge.net/projects/com0com/) (Signed Driver).
2. Создайте **три пары** виртуальных портов и соедините их в кольцо (см. схему в разделе «Физический уровень»).
3. Запустите три экземпляра:

```cmd
python main.py --node 1 --serial-left COM11 --serial-right COM12
python main.py --node 2 --serial-left COM21 --serial-right COM22
python main.py --node 3 --serial-left COM31 --serial-right COM32
```

Имена `COM*` замените на свои из настройки com0com. Параметры линии задаются в меню **Сеть → Параметры COM** (по умолчанию 115200 8N1).

### 6. Типичные проблемы на Windows

| Проблема | Решение |
|----------|---------|
| `python` не найден | Переустановите Python с галочкой PATH или используйте `py -3 demo.py` |
| Нет окна / ошибка tkinter | Переустановите Python с компонентом **tcl/tk** |
| Порт занят | Закройте другие программы, использующие COM |
| Кириллица в пути | Скопируйте проект в путь без проблемной кодировки, например `C:\kursovaya` |

## Руководство пользователя

Приложение предназначено для обмена письмами между тремя узлами, соединёнными в кольцо. Каждый узел имеет отдельное окно, два направления связи (**слева** и **справа**), папку **Входящие**, папку **Исходящие** и журнал сетевых событий.

### Запуск программы

Для демонстрации на одном компьютере запустите:

```cmd
python demo.py
```

После запуска откроются три окна: **Почта узла 1**, **Почта узла 2**, **Почта узла 3**. В демонстрационном режиме для каждого узла автоматически подставляются виртуальные COM-имена:

| Узел | COM слева | COM справа |
|------|-----------|------------|
| Узел 1 | `COM-1L` | `COM-1R` |
| Узел 2 | `COM-2L` | `COM-2R` |
| Узел 3 | `COM-3L` | `COM-3R` |

Эти COM-имена используются как программные заглушки и позволяют проверить работу без установки com0com. Если используются реальные COM-порты, выберите их в полях **COM слева** и **COM справа**.

### Настройка узла

В верхней части каждого окна расположен блок **Порты связи (соседи в кольце)**:

- **Слева** — соседний узел, подключённый к левому направлению связи;
- **Справа** — соседний узел, подключённый к правому направлению связи;
- **COM слева** и **COM справа** — виртуальные или реальные COM-порты;
- **Обновить COM** — повторно считывает список доступных COM-портов.

Для стандартного кольца оставьте значения по умолчанию:

| Текущий узел | Слева | Справа |
|--------------|-------|--------|
| Узел 1 | Узел 3 | Узел 2 |
| Узел 2 | Узел 1 | Узел 3 |
| Узел 3 | Узел 2 | Узел 1 |

Если выбран один и тот же сосед слева и справа, программа выведет предупреждение. Нельзя смешивать виртуальные COM-имена вида `COM-1L` / `COM-1R` с реальными портами ОС: нужно выбрать либо две виртуальные заглушки, либо два физических COM-порта.

### Подключение к сети

Для начала работы нажмите **Подключить** в каждом окне или выберите пункт меню **Сеть → Подключить канал**. При успешном подключении в строке состояния появится сообщение **Подключено**, а в журнале будет записано событие установки канала.

Пока канал отключён, кнопка **Написать** недоступна. Если попытаться создать письмо без подключения, программа выведет ошибку.

### Отправка письма

1. В окне отправителя нажмите **Написать** или выберите **Почта → Написать письмо...**.
2. В открывшейся форме выберите адресата: один из двух других узлов.
3. Заполните поле **Тема**.
4. Введите текст письма в поле **Текст**.
5. Нажмите **Отправить**.

После отправки письмо появится в таблице **Исходящие** у отправителя со статусом **отправлено**. Когда адресат получит письмо, статус изменится на **доставлено**.

### Получение и прочтение письма

Полученные письма отображаются в таблице **Входящие**. Чтобы открыть письмо:

1. Выберите строку письма во **Входящих**.
2. Нажмите **Открыть письмо** или дважды щёлкните по строке.
3. Ознакомьтесь с темой и текстом.
4. Нажмите **Отметить прочитанным и уведомить отправителя**.

После этого у получателя письмо получит статус **прочитано**, а отправитель увидит такой же статус в таблице **Исходящие**.

### Журнал событий

В нижней части окна находится **Протокол событий сети**. В него выводятся:

- подключение и закрытие портов;
- отправка письма;
- получение письма;
- подтверждение доставки;
- подтверждение прочтения;
- ошибки обработки или подключения.

Журнал нужен для контроля прохождения кадров по кольцу и демонстрации работы канального уровня.

### Закрытие портов и завершение работы

Чтобы разорвать соединение, нажмите **Закрыть порты** или выберите **Сеть → Закрыть порты**. После отключения статус окна изменится на **Отключено**, а выбор соседей и COM-портов снова станет доступен.

Для завершения работы закройте окна программы. При запуске через `demo.py` закрытие любого окна завершает все три узла.

## Как работает система: обзор

Три персональных компьютера соединены в **ненаправленное кольцо** (у каждого два соседа). Сообщения идут **по кольцу** от узла к узлу; промежуточные узлы **ретранслируют** кадры, если адресат не они.

```
        ┌─────────┐
        │ Узел 1  │
        └──┬───┬──┘
    left │   │ right
           │   │
    ┌──────┘   └──────┐
    │                 │
┌───┴───┐         ┌───┴───┐
│Узел 3 │         │Узел 2 │
└───┬───┘         └───┬───┘
    │                 │
    └────────┬────────┘
             │
      (замкнутое кольцо)
```

Данные проходят стек **сверху вниз** при отправке и **снизу вверх** при приёме:

```
[ GUI ]  ←→  Прикладной  ←→  Канальный  ←→  Физический  ←→  линия RS-232
```

## Работа приложения по модели OSI (7 уровней)

Ниже приведено соответствие реализованных модулей приложения уровням модели **OSI**. В учебной работе мы реализуем реальные механизмы **физического (L1)** и **канального (L2)** уровней, а логика «почты» относится к **прикладному (L7)**. Уровни **L3–L6** как отдельные подсистемы в проекте не выделяются (это нормально для локальной кольцевой лабораторной сети), но их роль можно описать, опираясь на поведение протокола.

### Сводная таблица соответствия

| Уровень OSI | Что это в нашей программе | Где в коде |
|-------------|---------------------------|------------|
| **L7 Прикладной** | Почтовая логика: письма, статусы, ACK прочтения/доставки, GUI | `src/application/*`, `src/ui/main_window.py` |
| **L6 Представления** | Формат данных: JSON + UTF‑8 (кодировка текста письма), сериализация/десериализация | `src/application/mail.py` |
| **L5 Сеансовый** | «Сеанс» в упрощённом виде: подключение/отключение канала и жизненный цикл узла | `src/physical/transport.py`, `src/node.py`, GUI (кнопки подключения) |
| **L4 Транспортный** | Надёжность не через TCP, а через подтверждения **ACK_DELIVERED/ACK_READ** на уровне кадров; повторная передача не реализована | `src/datalink/link_layer.py` |
| **L3 Сетевой** | Адресация узлов (1..3) и маршрутизация по кольцу (выбор направления, hop-limit) реализованы внутри L2-протокола | `src/datalink/link_layer.py` |
| **L2 Канальный** | Кадры, CRC‑16‑CCITT (циклический код), byte stuffing, отсечение дубликатов, пересылка по кольцу, служебные кадры ошибок | `src/common/frame.py`, `src/common/crc.py`, `src/datalink/link_layer.py` |
| **L1 Физический** | Два «порта» на узел (левый/правый), побайтовый ввод/вывод, параметры COM (baud, data bits, parity, stop bits), потоки чтения | `src/physical/transport.py`, `src/physical/ring_simulator.py` |

### L1. Физический уровень (Physical)

**Что реализовано:**
- Два направления связи для каждого узла: **left** и **right**.
- Два режима линии:
  - **виртуальная линия** (демо): обмен байтами через `VirtualPort` и `RingSimulator`;
  - **реальная линия**: открытие и чтение/запись через `pyserial`.
- Параметры линии COM:
  - **скорость** (baudrate),
  - **биты данных** (bytesize),
  - **чётность** (parity),
  - **стоп‑биты** (stopbits).

**Как работает:**
- При подключении `PhysicalLayer.connect()` открывает оба порта и запускает два фоновых потока чтения (по одному на сторону).
- Любые принятые байты передаются «наверх» через callback `on_bytes(data, side)` вместе с меткой стороны (`left/right`).

**Где в коде:**
- `src/physical/transport.py` — `PhysicalLayer`, циклы чтения `_read_loop_*`, метод `configure()` для параметров линии.
- `src/physical/ring_simulator.py` — `RingSimulator` и `VirtualPort` (эмуляция кольца).

---

### L2. Канальный уровень (Data Link)

**Что реализовано:**
- Формирование и разбор **кадров** с флагами `0x7E` (HDLC‑подобное кадрирование).
- **Byte stuffing** (экранирование) для `0x7E` и `0x7D`.
- **CRC‑16‑CCITT** (циклический контрольный код) для обнаружения ошибок.
- Адресация `SRC/DST`, тип кадра `TYPE`, ограничение обхода `HOP`, номер `SEQ`.
- Отсечение дубликатов по паре `(SRC, SEQ)` на коротком окне времени.
- Пересылка по кольцу: если `DST != текущий узел`, кадр пересылается на противоположный порт.
- Служебные кадры:
  - `ACK_DELIVERED` — подтверждение доставки;
  - `ACK_READ` — подтверждение прочтения;
  - `FRAME_ERROR` — служебная ошибка.

**Как работает:**
- Физический уровень отдаёт поток байт → `FrameDecoder` собирает кадры между флагами.
- `parse_frame()` проверяет CRC; если CRC неверен — кадр отбрасывается.
- `DataLinkLayer`:
  - принимает кадры, адресованные ему, и отдаёт payload прикладному уровню;
  - пересылает транзитные кадры с увеличением `HOP`.

**Дополнительная проверка совместимости параметров COM:**
- При подключении узел регистрирует свои параметры линии.
- При отправке/приёме письма выполняется проверка: если параметры у отправителя и получателя **разные**, передача запрещается и показывается ошибка.

**Где в коде:**
- `src/common/frame.py` — `Frame`, `encode_frame()`, `FrameDecoder`, `parse_frame()` (CRC/кадрирование).
- `src/common/crc.py` — `crc16_ccitt`.
- `src/datalink/link_layer.py` — маршрутизация, ACK, пересылка, защита от дубликатов.
- `src/physical/line_params.py` — реестр параметров линии и проверка совместимости узлов.

---

### L3. Сетевой уровень (Network)

В классической OSI‑модели L3 отвечает за маршрутизацию между сетями. В нашем проекте отдельного IP‑подобного уровня нет, но **функции L3** присутствуют как часть канального протокола:

- **адресация узлов** (DST/SRC = 1..3);
- **маршрутизация по кольцу** (выбор направления `left/right` к адресату);
- **ограничение зацикливания** (`HOP` и `MAX_HOPS`).

Где в коде: `src/datalink/link_layer.py` (методы `_pick_direction()`, `_forward()`).

---

### L4. Транспортный уровень (Transport)

Отдельного TCP/UDP‑аналога нет. Надёжность реализована в учебном виде через подтверждения:
- `ACK_DELIVERED` и `ACK_READ` подтверждают доставку и прочтение.

Ограничения:
- автоматической **повторной передачи** (retransmit) и контроля таймаутов нет — это упрощение для лабораторной.

Где в коде: `src/datalink/link_layer.py`, `src/application/app_layer.py`.

---

### L5. Сеансовый уровень (Session)

Сеансы в нашем приложении представлены жизненным циклом соединения:
- пользователь подключает канал (**Подключить**),
- работает (отправляет/читает письма),
- завершает сеанс (**Закрыть порты**).

Где в коде: `src/physical/transport.py` (`connect()/disconnect()`), `src/ui/main_window.py` (кнопки/меню).

---

### L6. Уровень представления (Presentation)

Вместо ASN.1/BER и т.п. используется простой формат данных:
- письмо и подтверждения сериализуются в **JSON**,
- текст кодируется в **UTF‑8**.

Где в коде: `src/application/mail.py` (`to_json()/from_json()`).

---

### L7. Прикладной уровень (Application)

Реализована услуга «почта»:
- создание письма (адресат, тема, текст),
- хранение **Входящих** и **Исходящих**,
- статусы письма: `sent` → `delivered` → `read`,
- обработка подтверждений и обновление статусов,
- интерфейс пользователя (GUI) + журнал событий.

Где в коде:
- `src/application/app_layer.py` — бизнес‑логика почты;
- `src/ui/main_window.py` — интерфейс (создание/просмотр письма, подключение, журнал).

## Физический уровень (`src/physical/`)

### Назначение

Имитирует работу интерфейса **RS-232C** и **нуль-модемного** соединения с соседями: открытие порта, настройка скорости, побайтовая передача и приём, поддержание канала.

### Состав

| Файл | Роль |
|------|------|
| `transport.py` | Класс `PhysicalLayer` — два порта (`left`, `right`), потоки чтения |
| `ring_simulator.py` | `RingSimulator` + `VirtualPort` — эмуляция кольца на одном ПК |

### Два порта на узле

Каждый ПК в кольце имеет **два соседа**:

- **`left`** — связь с предыдущим узлом в кольце (3→1, 1→2, 2→3);
- **`right`** — связь со следующим (1→2, 2→3, 3→1).

При **отправке** канальный уровень выбирает порт (`left` или `right`) в сторону адресата. При **приёме** физический уровень передаёт байты наверх вместе с меткой стороны (`"left"` / `"right"`), чтобы канальный уровень знал, в какой порт **ретранслировать** чужой кадр.

### Режимы работы

1. **Демо (`demo.py`, `--demo-ring`)**  
   `RingSimulator` связывает очереди TX/RX трёх узлов в кольцо:
   - `right(1) → left(2) → right(2) → left(3) → right(3) → left(1)` (по часовой);
   - `left(N) → right(N-1)` (против часовой).  
   Задержка передачи пропорциональна скорости (имитация времени на линии).

2. **Реальные COM (`pyserial`)**  
   `PhysicalLayer` открывает два последовательных порта (Windows `COM*`, macOS/Linux `/dev/*`) с параметрами, заданными пользователем: скорость, биты данных, чётность, стоп‑биты. Настройка — **Сеть → Параметры COM** (или кнопка **Параметры COM** на панели).

### Установка и разрыв канала

- **Сеть → Подключить канал** — `open()` на обоих портах, запуск потоков `_read_loop_*`.
- **Отключить** — закрытие портов, остановка потоков.
- Статус отображается в панели инструментов.

### Что передаётся

Только **сырые байты** — без понятия «письмо» или «кадр». Сборка кадров — задача канального уровня (`FrameDecoder` буферизует поток).

## Канальный уровень (`src/datalink/`)

### Назначение

Обеспечивает **логическую** передачу между узлами: кадрирование, **CRC-16-CCITT**, маршрутизация по кольцу, служебные кадры подтверждения и ошибок.

### Формат кадра

```
┌──────┬─────┬─────┬──────┬─────┬─────┬─────┬──────────┬────────┬──────┐
│ FLAG │ DST │ SRC │ TYPE │ HOP │ SEQ │ LEN │ PAYLOAD  │ CRC16  │ FLAG │
│ 0x7E │  1  │  1  │  1   │  1  │  2  │  2  │ 0..4096  │   2    │ 0x7E │
└──────┴─────┴─────┴──────┴─────┴─────┴─────┴──────────┴────────┴──────┘
```

| Поле | Описание |
|------|----------|
| **DST** | Адрес получателя (1, 2 или 3) |
| **SRC** | Адрес отправителя |
| **TYPE** | Тип кадра (см. ниже) |
| **HOP** | Счётчик переходов по кольцу (макс. 3 — защита от зацикливания) |
| **SEQ** | Номер кадра для отсечения дубликатов |
| **PAYLOAD** | Полезные данные (JSON письма или ACK) |
| **CRC16** | Контрольная сумма CCITT (полином 0x1021) по заголовку + payload |

Между флагами применяется **byte stuffing** (как в HDLC): байты `0x7E` и `0x7D` в теле кадра экранируются.

### Типы кадров (`src/common/constants.py`)

| Код | Имя | Назначение |
|-----|-----|------------|
| `0x01` | `FRAME_DATA` | Письмо (прикладной payload) |
| `0x02` | `FRAME_ACK_DELIVERED` | Подтверждение доставки |
| `0x03` | `FRAME_ACK_READ` | Подтверждение прочтения |
| `0x04` | `FRAME_ERROR` | Служебное сообщение об ошибке соседу |

### Отправка письма (узел 1 → узел 2)

1. Прикладной уровень формирует JSON письма.
2. `DataLinkLayer.send_letter()` упаковывает его в кадр `DATA`, `hop=0`, вычисляет CRC.
3. Выбирается направление: **кратчайший путь** по кольцу (`right` или `left`).
4. Кадр кодируется (`encode_frame`) и уходит в `PhysicalLayer.send()`.

### Ретрансляция (узел 3 пересылает кадр 1→2)

1. Кадр приходит на порт `left` или `right`.
2. `FrameDecoder` собирает кадр из потока байтов.
3. Если `DST ≠ мой узел` и `HOP < 3` — кадр пересылается на **противоположный** порт, `HOP` увеличивается на 1.
4. Если `DST = мой узел` — кадр обрабатывается локально, **не** ретранслируется.

### Защита от дубликатов и ошибок

- Пара `(SRC, SEQ)` запоминается на ~30 с; повторный кадр **отбрасывается** (защита от обхода кольца).
- При неверном CRC кадр **не разбирается** (в журнале можно увидеть отсутствие доставки).
- `FRAME_ERROR` информирует соседний узел о проблеме (по ТЗ).

### Подтверждения

| Событие | Кадр | Когда |
|---------|------|--------|
| Письмо принято получателем | `ACK_DELIVERED` | Сразу после успешного разбора `DATA` |
| Пользователь открыл письмо | `ACK_READ` | После нажатия «Прочитано» в GUI |

ACK маршрутизируются по кольцу так же, как `DATA`, до узла `SRC` исходного письма.

## Прикладной уровень (`src/application/`)

### Назначение

Реализует услугу **«почта»**: письма, адресация между тремя абонентами, статусы, уведомление отправителя о прочтении.

### Модель данных (`mail.py`)

**Письмо (`Letter`):**

| Поле | Описание |
|------|----------|
| `letter_id` | Уникальный идентификатор (8 символов UUID) |
| `sender` | Узел-отправитель (1..3) |
| `recipient` | Узел-получатель |
| `subject`, `body` | Тема и текст |
| `status` | `draft` → `sent` → `delivered` → `read` |

**Подтверждение (`AckPayload`):** `letter_id` + тип (`delivered` / `read`).

Сериализация — **JSON в UTF-8** внутри payload кадра.

### Логика (`app_layer.py`)

**Исходящие:**

1. Пользователь создаёт письмо → статус `sent`, запись в `outbox`.
2. Вызов `datalink.send_letter()`.
3. При `ACK_DELIVERED` → статус `delivered`.
4. При `ACK_READ` → статус `read` (отправитель видит «прочитано»).

**Входящие:**

1. Приём `DATA` → письмо в `inbox`, статус `delivered`.
2. Канальный уровень автоматически шлёт `ACK_DELIVERED` отправителю.
3. Пользователь открывает письмо → `open_letter()` → `ACK_READ` отправителю.

## Графический интерфейс (`src/ui/`)

Не является сетевым уровнем по ISO, но по ТЗ входит в **прикладной** уровень как «интерфейс с пользователем».

| Элемент | Функция |
|---------|---------|
| **Входящие** | Список полученных писем; двойной щелчок — просмотр и «прочитано» |
| **Исходящие** | Список отправленных; обновление статуса по ACK |
| **Протокол событий** | Лог сети: отправка, приём, ACK, ошибки |
| **Меню «Сеть»** | Подключение, параметры COM |
| **Меню «Почта»** | Создание письма, выбор адресата (1/2/3, кроме себя) |

Обновление GUI из фоновых потоков приёма — через `root.after(0, ...)` (потокобезопасность tkinter).

## Сборка узла (`src/node.py`)

Класс `NetworkNode` связывает уровни:

```
NetworkNode
├── PhysicalLayer   ← байты с линии
├── DataLinkLayer   ← кадры, маршрутизация
└── ApplicationLayer ← письма, статусы
         ↑
    NodeWindow (GUI)
```

Точка входа демо: `demo.py` — один процесс, общий `RingSimulator`, три окна `NodeWindow`.
